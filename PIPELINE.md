# Пайплайн написания экзаменационных билетов

Этот файл — главная инструкция для агента, когда пользователь просит:

- `напиши билеты для 1 лекции`;
- `сделай билеты по lecture-01`;
- `подготовь билет/билеты по лекции`;
- похожую формулировку про генерацию, ревью или финализацию билетов.

Цель пайплайна — быстро перейти от номера лекции к готовым материалам в `tickets/`, не тратя контекст на повторное восстановление правил.

## Главный принцип

Билеты пишутся не как общий конспект лекции, а строго по экзаменационным вопросам из:

```text
csa-rolling/exam-questions-blitz.md
```

Один экзаменационный вопрос — один раздел `## Билет N. ...` в `tickets/lecture-XX.md`.

Заголовок билета должен повторять формулировку вопроса по смыслу, лучше дословно. Если вопрос состоит из нескольких частей, ответ обязан закрывать каждую часть. Например, для вопроса:

```text
Что такое "вычислительная платформа" с точки зрения пользователя? Каков её состав?
```

нельзя писать общий рассказ про всю первую лекцию. Нужно прямо ответить:

1. что такое вычислительная платформа с точки зрения пользователя;
2. из каких частей она состоит;
3. какими примерами это объяснял лектор.

## Источники фактов

Факты для билета брать только из источников конкретной лекции:

1. `sources/lecture-XX/my-notes.md`;
2. `sources/lecture-XX/slides.md`;
3. `sources/lecture-XX/transcript.cleaned.md`;
4. `sources/lecture-XX/transcript.raw.md`.

`sources/lecture-XX/source-pack.md` использовать как компактную карту лекции, если он уже собран, но спорные места сверять с файлами выше.

`csa-rolling/exam-questions-blitz.md` использовать как источник формулировок вопросов, а не как источник фактов.

`csa-rolling/course-note/*`, `csa-rolling/README.md`, `csa-rolling/exam.md` и другие файлы курса можно использовать только как ориентир или для проверки структуры, но не как источник фактов для ответа, если эти факты не подтверждены источниками лекции. Если приходится использовать внешний/дополнительный материал, вынести его в отдельный блок `[внешнее пояснение]`.

Если факта нет в источниках лекции, писать `[нет в источниках]`. Если место сомнительное, писать `[проверить]`. Нельзя заполнять пробелы догадками.

## Быстрый запуск по запросу пользователя

Когда пользователь пишет `напиши билеты для 1 лекции`, агент должен:

1. Преобразовать номер в `lecture-01`.
2. Найти запись лекции в `lectures.yaml`.
3. Открыть `PIPELINE.md`, `AGENTS.md` и нужные файлы из `sources/lecture-01/`.
4. Найти вопросы лекции 1 в `csa-rolling/exam-questions-blitz.md`.
5. Проверить, готовы ли источники:
   - есть ли настоящий `transcript.raw.md`, а не заглушка;
   - есть ли настоящий `transcript.cleaned.md`, а не заглушка;
   - есть ли содержательный `slides.md`;
   - есть ли `source-pack.md`.
6. Если нужный подготовительный файл отсутствует или является заглушкой, сначала выполнить соответствующий этап пайплайна.
7. Сгенерировать или обновить `tickets/lecture-01.md` строго по экзаменационным вопросам.
8. Провести ревью билета по источникам.
9. Исправить билет по ревью.
10. Создать или обновить `tickets/lecture-01.cram.md`.
11. В конце сообщить, что изменено, что проверено и какие `[проверить]` остались.

Если пользователь явно просит только ревью, анализ или план, не выполнять запись файлов без отдельного запроса.

## Как определить вопросы лекции

Основной список вопросов находится в `csa-rolling/exam-questions-blitz.md`.

В файле встречаются маркеры вида:

```markdown
1. <!-- 1 --> Что такое ...
...
1. <!-- 2 --> Почему ...
```

Маркер `<!-- 1 -->` означает начало блока вопросов лекции 1. Вопросы лекции продолжаются до следующего маркера другой лекции. Для `lecture-01` нужно брать вопросы от `<!-- 1 -->` до `<!-- 2 -->`.

Если маркер указан как `<!-- 15-16 -->`, вопрос относится к нескольким лекциям. В таком случае включать его в билет только если источники текущей лекции действительно покрывают этот вопрос; иначе пометить `[проверить]` или оставить вопрос для другой лекции.

Если соответствие вопроса лекции неочевидно, не угадывать молча: отметить `[проверить]` и объяснить в статусе подготовки.

## Подготовка источников

### 1. Raw transcript

Если `sources/lecture-XX/transcript.raw.md` — заглушка, попробовать получить транскрипт:

```bash
python3 scripts/fetch_transcript.py --lecture lecture-XX
```

Если транскрипт получить не удалось, зафиксировать это в логе/ответе и не выдумывать объяснения лектора.

### 2. Slides

Если `sources/lecture-XX/slides.md` уже импортирован из `csa-rolling`, использовать его.

Если есть PDF-слайды и `slides.md` не готов, использовать:

```bash
python3 scripts/extract_slides.py --lecture lecture-XX
```

При обработке слайдов сохранять определения, схемы, таблицы, формулы и учебные картинки. Декоративные картинки не превращать в учебные утверждения.

### 3. Cleaned transcript

Очищать `transcript.raw.md` по `prompts/clean-transcript.md`.

Правила:

- не менять смысл;
- не добавлять факты;
- убрать мусор автосубтитров;
- сохранить термины, числа, формулы, названия команд, регистров, блоков и архитектур;
- сомнительное помечать `[проверить]`.

### 4. Source pack

Собрать `sources/lecture-XX/source-pack.md` по `prompts/build-source-pack.md`.

`source-pack.md` должен содержать:

- основные темы лекции;
- ключевые определения;
- механизмы и идеи;
- формулы;
- схемы и картинки;
- примеры из лекции;
- возможные экзаменационные вопросы;
- неясные места.

Для важных утверждений указывать источник в скобках: `(my-notes)`, `(slides)`, `(transcript.cleaned)`, `(transcript.raw)`.

## Генерация `tickets/lecture-XX.md`

Создавать билет по `prompts/generate-ticket.md`, но с обязательным уточнением:

```text
Список билетов берётся из блока lecture-XX в csa-rolling/exam-questions-blitz.md.
```

### Стиль устного ответа

`Короткий ответ` — это не мини-конспект. Его задача — дать студенту минимальную когнитивную нагрузку перед устным ответом.

Правила:

- писать 2-4 коротких предложения, максимум 5 для сложного многосоставного вопроса;
- начинать сразу с ответа по существу, без вводных вроде "в первой лекции", "как было сказано в лекции", "материалы показывают";
- не перегружать короткий ответ длинными перечислениями, если этот список уже есть на картинке или будет раскрыт ниже;
- если нужен состав/уровни/этапы, в коротком ответе дать суть и 1-2 примера, а подробное перечисление вынести в `Подробный ответ`;
- не использовать сложные словосочетания без пользы для студента; если термин нужен, тут же дать простое пояснение или перенести его в подробный ответ;
- примеры должны помогать понять идею, а не выглядеть как набор умных слов.

Хорошая форма короткого ответа:

```markdown
Вычислительная платформа — это набор инструментов, который позволяет описать вычислительный процесс и запустить его.
Примеры: shell, язык программирования, фреймворк или система команд процессора.
В составе платформы важно выделить язык, модель вычислений, данные/типы и базовые операции.
```

Структура файла:

```markdown
# Lecture XX — Название лекции

## Источники

- `sources/lecture-XX/source-pack.md`
- `sources/lecture-XX/my-notes.md`
- `sources/lecture-XX/slides.md`
- `sources/lecture-XX/transcript.cleaned.md`
- `sources/lecture-XX/transcript.raw.md`
- `csa-rolling/exam-questions-blitz.md` — только формулировки вопросов

## Список билетов

1. Точная формулировка вопроса
2. ...

---

## Билет 1. Точная формулировка вопроса

### Короткий ответ

2-4 коротких предложения, которые прямо отвечают на вопрос и не требуют перечитывания.

### Схема / картинка

Полезная схема из слайдов или `[подходящей картинки в слайдах не найдено]`.

### Подробный ответ

Последовательное объяснение того, что спросили в вопросе.

### Ключевые определения

- **Термин** — определение из источников.

### Пример

Пример из лекции/слайдов или `[пример не найден в источниках]`.

### Возможные вопросы преподавателя

- ...

### Что обязательно запомнить

- ...

### Проверить

- `[проверить]` ...
- `[нет в источниках]` ...
```

Внутри `Подробного ответа` можно использовать подзаголовки, совпадающие с частями вопроса. Например:

```markdown
#### Что это с точки зрения пользователя

#### Состав вычислительной платформы

#### Примеры из лекции
```

Это помогает не уйти в общий конспект вместо ответа на вопрос.

Если в вопросе есть уровни, состав, этапы или элементы схемы, в `Подробном ответе` нужно кратко пояснить каждый важный элемент простыми словами. Не ограничиваться фразой "они перечислены на картинке".

## Ревью

После генерации обязательно провести ревью по `prompts/review-ticket.md`.

Проверить:

- каждый билет отвечает именно на свою формулировку из `exam-questions-blitz.md`;
- все части вопроса закрыты;
- нет фактов без опоры на источники лекции;
- нет домыслов и внешних знаний без блока `[внешнее пояснение]`;
- определения не противоречат слайдам;
- объяснения лектора подтверждены транскриптом;
- картинки уместны и стоят сразу после `Короткий ответ`;
- `Короткий ответ` простой, короткий, без лишней ссылки на номер лекции и без перегруженных перечислений;
- все `[проверить]` и `[нет в источниках]` вынесены в раздел `Проверить`;
- ответ реально можно произнести устно.

Ревью можно хранить в ответе пользователю или в логе, если пользователь просит. Финальный билет нельзя считать готовым без ревью.

## Финализация

Финализировать по `prompts/finalize-ticket.md`:

1. Исправить ошибки из ревью.
2. Убрать неподтвержденные утверждения.
3. Добавить пропущенные части вопросов, если они есть в источниках.
4. Оставить `[нет в источниках]`, если подтвердить нельзя.
5. Создать или обновить `tickets/lecture-XX.cram.md`.
6. Добавить в конец `tickets/lecture-XX.md` статус подготовки.

Формат статуса:

```markdown
---

## Статус подготовки

- статус: `reviewed` / `needs-check`
- дата финализации: `YYYY-MM-DD`
- оставшиеся проверки:
  - ...
```

`reviewed` ставить только если все существенные утверждения подтверждены источниками и нет критичных `[проверить]`.

`needs-check` ставить, если:

- транскрипт автоматический и есть сомнительные места;
- источник противоречив;
- вопрос из экзаменационного списка покрыт лекцией не полностью;
- не хватает данных для уверенного ответа.

## Краткая версия `lecture-XX.cram.md`

Краткая версия не должна содержать новых фактов. Она сжимает уже финализированный билет.

Структура:

```markdown
# Lecture XX — краткое повторение

## Главное

- ...

## Определения

- **Термин** — определение.

## Формулы

$$
...
$$

## Типовые вопросы

- ...

## Нужно проверить

- `[проверить]` ...
- `[нет в источниках]` ...
```

Если формул нет, писать `[формул в источниках не найдено]`.

## Самопроверка перед ответом пользователю

Перед финальным ответом агент обязан проверить:

- `tickets/lecture-XX.md` создан или обновлён;
- `tickets/lecture-XX.cram.md` создан или обновлён;
- билеты соответствуют вопросам из `exam-questions-blitz.md`;
- в каждом билете закрыты все части вопроса;
- факты подтверждены источниками лекции;
- нет неподписанных внешних знаний;
- сомнительные места помечены `[проверить]`;
- проведено ревью;
- пользователь получил краткий список изменений и оставшихся рисков.
