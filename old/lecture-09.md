## Билет 1. Что такое стековые системы команд? Каковы основные элементы? Что такое неявная адресация аргументов? Каковы их достоинства и недостатки?

### Короткий ответ

#### Что такое стековые системы команд?

Стековая система команд обрабатывает данные не в регистрах общего назначения, а на стеке.
Операция берёт значения с вершины стека и кладёт результат обратно.

#### Каковы основные элементы?

Основные элементы — data stack, return stack, операции `push/pop`, чтение `@`, запись `!` и переходы через стек.

#### Что такое неявная адресация аргументов?

Неявная адресация означает, что инструкция не пишет адреса операндов явно.
Она предполагает, что операнды лежат на вершине стека.

#### Каковы их достоинства и недостатки?

Достоинства — короткие инструкции, простая ISA и удобные процедуры.
Недостатки — трудный ILP, проблемы с большим объёмом данных и непривычная модель программирования.

### Схема / картинка

![Схема стековой архитектуры](../assets/course-fig/proc-stack-arch-scheme-only.png)

Схема показывает, что вычисления идут вокруг стека, а не вокруг явных регистровых операндов.

### Подробный ответ

#### Что такое стековые системы команд?

В стековой ISA основная работа идёт через стек.
Например, `mul` можно понимать как:

```text
a = pop()
b = pop()
c = a * b
push(c)
```

Программе не нужно явно указывать, где лежат `a` и `b`.
Архитектура сама предполагает вершину стека.

#### Каковы основные элементы?

Основные элементы:

- стек данных для операндов и результатов;
- стек возвратов для процедур и адресов возврата;
- операции загрузки на стек и снятия со стека;
- операции чтения `@` и записи `!` через стек;
- переходы, где условие или адрес могут браться со стека.

#### Что такое неявная адресация аргументов?

Неявная адресация — это адресация без явного поля операнда.
Команда `+` не говорит "сложить `r1` и `r2`".
Она просто берёт два верхних элемента стека.
Поэтому команда короче: в ней часто остаётся только opcode.

#### Каковы их достоинства и недостатки?

Достоинства:

- короткие инструкции;
- простая система команд;
- простой компилятор;
- удобные процедуры, рекурсия и автоматическая память;
- хорошая локальность при работе с верхушкой стека.

Недостатки:

- большой набор данных на стеке приходится постоянно перемешивать;
- динамические структуры всё равно требуют памяти;
- параллелизм уровня инструкций труден, потому что команды завязаны на одно состояние стека;
- модель непривычна для программистов регистровых машин.

### Ключевые определения

- **Stack Machine** — архитектура, где операнды и результаты проходят через стек.
- **Неявная адресация** — операнды не записаны в инструкции, а подразумеваются архитектурой.
- **Data stack** — стек данных.
- **Return stack** — стек возвратов.

### Пример

`A @ B @ * C @ + Y !` читает значения через `@`, выполняет операции над стеком и записывает результат через `!`.

### Возможные вопросы преподавателя

- Почему стековые инструкции короткие?
- Почему стек мешает параллельному исполнению?
- Чем стековая архитектура отличается от аккумуляторной?

### Что обязательно запомнить

- Операнды лежат на стеке.
- Многие команды имеют ноль явных операндов.
- Плюсы: компактность и процедуры.
- Минусы: ILP и большие данные.

## Билет 2. Какие виды стеков есть в стековых процессорах? Их количества?

### Короткий ответ

В стековых процессорах обычно выделяют стек данных и стек возвратов.
Стек данных хранит операнды и промежуточные результаты.
Стек возвратов хранит адреса возврата и помогает процедурам.
В учебной архитектуре из источников есть два стека: data stack и return stack.
В реальных архитектурах стеков может быть больше.
Несколько стеков могут упростить переключение потоков.
Стек часто реализуется аппаратно, чтобы быстро читать верхние элементы.

### Схема / картинка

![Классы стеков](../assets/course-fig/proc-stack-classes.png)

Картинка показывает варианты организации стеков в стековых процессорах.

### Подробный ответ

Стековый процессор не обязан иметь ровно один стек.
В источниках явно названы два важных вида.

**Data stack** хранит данные вычислений.
Арифметические инструкции берут операнды с него и возвращают результат туда же.

**Return stack** используется для процедур.
Он помогает хранить адреса возврата и обслуживать вложенные вызовы.
Это важно для процедурного стиля, рекурсии и high-level language computer.

Количество стеков может различаться.
В учебной архитектуре, с которой студенты работают в лабораторной, используются два стека.
В реальных процессорах стеков может быть несколько.
Лектор объясняет, что несколько стеков могут дать дешёвую поддержку нескольких потоков: процессор переключается между стековыми состояниями.

Стек обычно не делают просто областью обычной памяти.
Для скорости верхушка стека реализуется аппаратно, чтобы за один такт читать top-of-stack и second-of-stack.

### Ключевые определения

- **Data stack** — стек операндов и результатов.
- **Return stack** — стек адресов возврата и служебной информации процедур.
- **Top of stack** — верхний элемент стека.

### Пример

При вызове рекурсивного `fac` аргументы и промежуточные значения лежат на data stack, а возвраты из вызовов поддерживаются frame/return stack.

### Возможные вопросы преподавателя

- Зачем отделять data stack от return stack?
- Почему стек выгодно реализовывать аппаратно?
- Как несколько стеков могут помочь потокам?

### Что обязательно запомнить

- Минимально важны data stack и return stack.
- В учебной архитектуре их два.
- В реальных системах может быть больше.
- Верхушка стека должна быть быстрой.

## Билет 3. Каковы основные принципы описания алгоритмов для стековых процессоров? Что такое Forth и High-level language computer?

### Короткий ответ

#### Каковы основные принципы описания алгоритмов для стековых процессоров?

Алгоритм для стекового процессора описывается как последовательность действий над стеком.
Данные сначала кладутся на стек, потом операции снимают аргументы и возвращают результат.
Процедуры получают аргументы через стек и оставляют результат на стеке.

#### Что такое Forth и High-level language computer?

Forth — императивный стековый язык программирования.
Он поддерживает процедуры, рекурсию, структурное программирование и расширяемость.
High-level language computer — процессор, где конструкции высокого уровня естественно поддержаны архитектурой.

### Схема / картинка

![Forth factorial evaluation](../assets/course-fig/proc-stack-evaluation-example.png)

Схема показывает вычисление факториала через data stack и frame.

### Подробный ответ

#### Каковы основные принципы описания алгоритмов для стековых процессоров?

Стековый алгоритм пишется в стиле обратной польской записи.
Сначала кладутся операнды, затем вызывается операция.
Например, выражение:

```text
X = (A + B) * (C + D)
```

может быть представлено как:

```text
A B + C D + *
```

Процедуры тоже работают через стек.
Вызывающая часть кладёт аргументы.
Процедура забирает нужные значения и оставляет результат.
Важно, чтобы процедура не снимала со стека лишнего.

#### Что такое Forth и High-level language computer?

Forth — стековый язык.
Он позволяет определять новые команды, использовать процедуры и рекурсию.
Синтаксис может выглядеть непривычно, но он близок к тому, как стековый процессор реально исполняет операции.

High-level language computer — идея, что архитектура процессора естественно поддерживает конструкции языков высокого уровня.
В стековой архитектуре это процедуры, автоматическая память, рекурсия и выражения.
Поэтому часть того, что в регистровых машинах требует соглашений о вызовах и управления регистрами, здесь получается проще.

### Ключевые определения

- **Forth** — императивный язык на основе стека.
- **Frame** — состояние вложенных вызовов/возвратов в трассе процедуры.
- **High-level language computer** — архитектура, поддерживающая высокоуровневые конструкции на уровне ISA.

### Пример

Рекурсивная функция `fac` в Forth берёт число со стека, проверяет условие, вызывает себя и умножает результат.

### Возможные вопросы преподавателя

- Почему стековая запись похожа на обратную польскую?
- Почему процедуры дешевле на стековой машине?
- Почему модель непривычна для программиста регистровых машин?

### Что обязательно запомнить

- Данные идут через стек.
- Операции часто без аргументов.
- Forth — стековый язык.
- High-level language computer поддерживает процедуры и рекурсию архитектурно.

## Билет 4. Каковы источники роста производительности процессоров?

### Короткий ответ

Источники роста производительности процессоров — частота, специализация, параллелизм, адаптация структуры и динамическая адаптация.
Рост частоты ускоряет такты, но ограничивается памятью и физикой.
Специализация добавляет аппаратные блоки или инструкции под частые задачи.
Параллелизм бывает уровня битов, инструкций и задач.
Адаптация структуры вычислителя под задачу повышает эффективность.
Динамическая адаптация меняет аппаратную конфигурацию под текущую задачу.
Например, аппаратный умножитель быстрее программного умножения.
Но каждая специализация уменьшает универсальность.

### Схема / картинка

![Рост производительности](../assets/course-fig/proc-performance-grow.jpg)

График показывает исторические тренды частоты, числа ядер, транзисторов и производительности.

### Подробный ответ

В источниках названы пять источников роста.

**Частота.**
Если тактовая частота растёт, процессор может выполнять больше шагов в секунду.
Но рост не идеально линейный, потому что память и внешнее окружение могут не успевать.

**Специализация системы команд и аппаратуры.**
Можно добавить специальную инструкцию или аппаратный блок.
Например, аппаратный умножитель быстрее программного цикла умножения.

**Параллелизм.**
Параллелизм уровня битов обрабатывает больше битов за раз.
Параллелизм уровня инструкций запускает несколько стадий или инструкций одновременно.
Параллелизм уровня задач использует несколько потоков или вычислителей.

**Адаптация структуры под задачу.**
Если вычислитель физически ближе к структуре алгоритма, он может работать быстрее и экономичнее.
Цена — потеря гибкости.

**Динамическая адаптация.**
Аппаратура может менять конфигурацию под текущую задачу.
В источниках примером выступает реконфигурируемая логика, которую можно прошивать под нужный алгоритм.

### Ключевые определения

- **Параллелизм уровня битов** — одновременная обработка нескольких битов.
- **Параллелизм уровня инструкций** — одновременное выполнение частей разных инструкций.
- **Динамическая адаптация** — изменение структуры вычислителя во время работы или под задачу.

### Пример

Полностью аппаратный умножитель ускоряет умножение по сравнению с программной реализацией через несколько шагов.

### Возможные вопросы преподавателя

- Почему частота не даёт бесконечного роста?
- Чем специализация отличается от универсальности?
- Какие уровни параллелизма названы в лекции?

### Что обязательно запомнить

- Частота, специализация, параллелизм, адаптация, динамическая адаптация.
- Производительность ограничивается памятью, теплом и распараллеливаемостью.
- Специализация почти всегда стоит гибкости.

## Билет 5. Какие законы и ограничения влияют на производительность современных процессоров?

### Короткий ответ

На производительность современных процессоров влияют закон Мура, закон Амдала, закон Деннарда, Power Wall и Memory Wall.
Закон Мура описывает рост числа транзисторов.
Закон Амдала ограничивает выигрыш от параллелизма.
Закон Деннарда связан с масштабированием транзисторов и мощности.
Power Wall ограничивает рост из-за мощности и отвода тепла.
Memory Wall ограничивает процессор скоростью памяти.
Эти ограничения объясняют, почему нельзя просто повышать частоту, добавлять ядра и уменьшать транзисторы бесконечно.

### Схема / картинка

![Рост производительности](../assets/course-fig/proc-performance-grow.jpg)

График собирает исторические тренды и показывает замедление прежних источников роста.

### Подробный ответ

Современные процессоры упираются не в одно ограничение, а в набор ограничений.

**Закон Мура** долго описывал рост числа транзисторов.
Но это наблюдение, а не физический механизм, который гарантирует рост.

**Закон Амдала** показывает предел параллельного ускорения.
Если часть задачи не распараллеливается, добавление вычислителей даёт всё меньший эффект.

**Закон Деннарда** связывает уменьшение транзисторов с сохранением плотности мощности.
Когда масштабирование перестаёт работать, частоту и плотность нельзя повышать прежними темпами.

**Power Wall** говорит, что мощность и тепло становятся стеной.
Даже если транзисторы есть, их нельзя все активно включить.

**Memory Wall** говорит, что процессор становится быстрее памяти.
Если данные не приходят вовремя, вычислительные блоки простаивают.

### Ключевые определения

- **Закон Мура** — эмпирическое наблюдение о росте числа транзисторов.
- **Закон Амдала** — ограничение ускорения из-за непараллелизуемой части.
- **Закон Деннарда** — масштабирование мощности при уменьшении MOSFET.
- **Power Wall** — ограничение по мощности/теплу.
- **Memory Wall** — ограничение по доступу к памяти.

### Пример

Удвоение частоты не даёт удвоения производительности, если процессор большую часть времени ждёт память.

### Возможные вопросы преподавателя

- Какое ограничение связано с памятью?
- Какое ограничение связано с теплом?
- Почему закон Мура не является объяснением?

### Что обязательно запомнить

- Мур — транзисторы.
- Амдал — параллелизм.
- Деннар — масштабирование мощности.
- Power Wall — тепло.
- Memory Wall — память.

## Билет 6. Что представляют собой закон Мура и закон Деннарда?

### Короткий ответ

#### Что представляет собой закон Мура?

Закон Мура — наблюдение, что число транзисторов в плотной интегральной схеме удваивается примерно каждые два года.
Это эмпирический тренд, а не физическая причина роста.

#### Что представляет собой закон Деннарда?

Закон Деннарда говорит, что при уменьшении транзисторов плотность мощности примерно сохраняется.
Напряжение и ток уменьшаются вместе с размером.
Это позволяло повышать плотность и частоту без пропорционального перегрева.
Сейчас масштабирование упирается в физические размеры, токи утечки и тепло.

### Схема / картинка

![Dennard scaling](../assets/course-fig/dennard-scaling.jpg)

Картинка показывает идею масштабирования и остановку прежнего тренда.

### Подробный ответ

#### Что представляет собой закон Мура?

Закон Мура описывает историческое наблюдение: количество транзисторов на плотной интегральной схеме растёт примерно вдвое за два года.
В лекции подчёркнуто, что это именно наблюдение.
Оно показывает тренд индустрии, но само не объясняет, почему процессоры становятся быстрее.

Когда технологическое уменьшение замедляется, закон Мура перестаёт работать как прежний ориентир роста.

#### Что представляет собой закон Деннарда?

Закон Деннарда относится к масштабированию MOSFET.
При уменьшении транзисторов напряжение и ток тоже должны уменьшаться, а плотность мощности оставаться примерно постоянной.
Это позволяло делать транзисторы меньше и быстрее без постоянного роста тепла на единицу площади.

Практические проблемы:

- транзисторы становятся слишком малы физически;
- производство становится сложным и дорогим;
- растут токи утечки;
- тепло трудно отводить с плотной площади.

Отсюда возникают Power Wall и Dark Silicon.

### Ключевые определения

- **Закон Мура** — эмпирический тренд удвоения числа транзисторов примерно каждые два года.
- **Закон Деннарда** — закон масштабирования, связывающий размер транзистора, напряжение, ток и плотность мощности.
- **Токи утечки** — нежелательное потребление, возникающее при малых размерах и уходящее в тепло.

### Пример

Переход к меньшему техпроцессу раньше позволял повышать частоту и уменьшать потребление, но на современных малых размерах этот путь ограничен.

### Что обязательно запомнить

- Мур — "транзисторов больше".
- Деннар — "меньше транзистор, но плотность мощности примерно та же".
- Оба тренда сейчас ограничены физикой и производством.

## Билет 8. Что такое Power Wall и тёмный кремний?

### Короткий ответ

#### Что такое Power Wall?

Power Wall — это ограничение роста производительности мощностью и тепловыделением.
Процессор нельзя бесконечно ускорять, если тепло невозможно отвести.

#### Что такое тёмный кремний?

Тёмный кремний — это часть кристалла, которую нельзя держать активной одновременно с остальной.
Транзисторы физически есть, но их приходится выключать или использовать по очереди.
Причина — ограничение по температуре и плотности мощности.
Поэтому дизайн чипа включает не только логику, но и тепловое размещение.

### Схема / картинка

![Dark silicon](../assets/course-fig/dark-silicon.jpg)

Картинка иллюстрирует, что активная площадь ограничивается возможностью отвести тепло.

### Подробный ответ

#### Что такое Power Wall?

Power Wall появляется, когда дальнейший рост частоты, плотности транзисторов и активности схемы упирается в мощность.
Электрическая энергия превращается в тепло.
Если тепло не отводится, чип перегревается и не может работать надёжно.

Это связано с остановкой прежнего масштабирования.
Транзисторы стали меньше, но токи утечки и плотность тепла стали серьёзной проблемой.

#### Что такое тёмный кремний?

Dark Silicon означает, что часть кристалла остаётся "тёмной", то есть неактивной.
Её нельзя включить одновременно с остальной активной логикой без перегрева.

На слайдах есть идея размещения тёмных участков как теплового паттерна.
Если активные блоки собрать слишком плотно, температура растёт.
Если распределить активные и неактивные области, тепловой профиль улучшается.

### Ключевые определения

- **Power Wall** — предел производительности из-за мощности и тепла.
- **Dark Silicon** — неактивная часть кристалла, нужная из-за тепловых ограничений.
- **Тепловой профиль** — распределение температуры по кристаллу.

### Пример

На более мелком техпроцессе тот же функционал можно уместить в меньшую площадь, но нельзя использовать всю плотную площадь одновременно из-за отвода тепла.

- Почему тепловое размещение стало частью проектирования?

### Что обязательно запомнить

- Power Wall — стена мощности и тепла.
- Dark Silicon — транзисторы есть, но часть должна молчать.
- Чем плотнее чип, тем важнее тепло.

## Билет 9. Какие подходы существуют к решению проблемы Memory Wall в рамках процессоров семейства фон Неймана?

### Короткий ответ

Memory Wall — это разрыв между скоростью процессора и скоростью памяти.
В рамках фон-неймановских процессоров проблему смягчают несколькими путями.
Первый путь — кеширование и иерархия памяти.
Второй путь — разделение потоков инструкций и данных, например Гарвардская или модифицированная Гарвардская организация.
Третий путь — увеличение каналов или портов доступа к памяти.
Четвёртый путь — оптимизация доступа к данным, чтобы реже ждать память.
Но полностью снять проблему трудно, потому что растут сложность, стоимость и вопросы консистентности.

### Схема / картинка

![Memory Wall](../assets/course-fig/memory-wall-cpu-vs-memory.png)

График показывает растущий разрыв между производительностью CPU и памяти.

### Подробный ответ

Memory Wall означает, что вычислительные блоки могут быть быстрее, чем поставка данных из памяти.
Процессор простаивает, если не получает операнды вовремя.
В лекции это связывается с историческим разрывом между ростом CPU performance и memory performance.

Подходы смягчения:

**Кеширование и иерархия памяти.**
Кеши приближают часто используемые данные к процессору.
Подробно эта тема раскрывается в последующих лекциях, но в lecture-09 она названа как важный трюк.

**Разделение доступа к инструкциям и данным.**
Гарвардская архитектура частично решает проблему, потому что разделяет каналы.
Модифицированный Гарвард с I-cache и D-cache тоже помогает конвейеру.

**Больше каналов или портов.**
Можно пытаться увеличить число путей к памяти.
Но это быстро усложняет программирование, аппаратную реализацию и консистентность памяти.

**Оптимизация доступа к данным.**
Нужно так организовать алгоритм и микроархитектуру, чтобы реже ходить в медленную память.
Это включает регистры, кеши и локальность доступа.

### Ключевые определения

- **Memory Wall** — ограничение производительности из-за разрыва CPU и памяти.
- **Кеш** — быстрая память между процессором и основной памятью.
- **Консистентность памяти** — согласованность наблюдаемых значений при нескольких путях доступа.

### Пример

Если процессор ускорили, но память осталась прежней, программа с частыми обращениями к памяти не ускорится пропорционально.

### Возможные вопросы преподавателя

- Почему Гарвардская архитектура только частично помогает?
- Почему нельзя бесконечно добавлять каналы памяти?
- Как кеш связан с Memory Wall?

### Что обязательно запомнить

- Memory Wall — процессор ждёт данные.
- Кеши и иерархия памяти помогают.
- Разделение инструкций и данных помогает.
- Больше каналов создаёт проблему сложности и согласованности.

## Билет 10. Что такое RISC? Каковы особенности и предпосылки появления (ПО и аппаратура)?

### Короткий ответ

#### Что такое RISC?

RISC — это подход с сокращённым и однообразным набором часто используемых инструкций.
Инструкции кодируются просто, чтобы их было легче декодировать и конвейеризировать.

#### Каковы особенности и предпосылки появления?

Сложные операции встречаются редко и заменяются группами простых команд.
Появились языки высокого уровня, поэтому удобство ручного ассемблера стало менее важным.
Простой декодер освобождает место для регистров и кеша.
Главная цель — ускорить поток частых инструкций, а не одну сложную команду.

### Схема / картинка

![RISC](../assets/course-fig/risc.png)

Картинка используется в блоке определения RISC.

### Подробный ответ

#### Что такое RISC?

RISC расшифровывается как Reduced Instruction Set Computer.
Это подход к проектированию, где набор инструкций сокращён до часто используемых и единообразно закодированных.
Упрощение нужно не ради бедности ISA, а ради более быстрой реализации.

#### Каковы особенности и предпосылки появления?

Предпосылки со стороны программного обеспечения:

- программисты стали чаще писать на языках высокого уровня;
- компилятор может генерировать серии простых команд;
- удобство сложного ассемблера стало менее важным.

Предпосылки со стороны аппаратуры:

- сложные инструкции редки;
- сложные операции можно заменить простыми;
- единый формат упрощает декодер;
- вместо памяти микрокода можно использовать регистры и кеш;
- однообразные инструкции хорошо ложатся на pipeline.

RISC оптимизирует не отдельную инструкцию, а поток инструкций.
Аналогия из лекции — конвейерное производство: одна единица может проходить дольше, но поток в целом даёт больше результатов.

### Ключевые определения

- **RISC** — подход с простыми часто используемыми инструкциями и единым форматом.
- **Pipeline** — конвейерное исполнение инструкций.
- **Load-store** — разделение операций памяти и регистровой арифметики.

### Пример

Одна CISC-инструкция может заменяться четырьмя RISC-инструкциями.
Но RISC выигрывает, если эти инструкции быстро декодируются и идут по конвейеру.

### Возможные вопросы преподавателя

- Почему появление языков высокого уровня помогло RISC?
- Почему единый формат важен для декодера?
- Почему RISC не означает "мало инструкций любой ценой"?

### Что обязательно запомнить

- RISC упрощает инструкции.
- Главная выгода — pipeline и быстрый поток.
- ПО стало высокоуровневым.
- Аппаратуре стало проще декодировать и оптимизировать команды.

## Билет 11. Как сравниваются CISC и RISC с точки зрения архитектуры?

### Короткий ответ

CISC с точки зрения архитектуры даёт сложные и выразительные инструкции.
Одна инструкция может совмещать доступ к памяти, вычисление и запись результата.
RISC с точки зрения архитектуры даёт простые однообразные инструкции.
Операции с памятью отделены от операций над регистрами.
CISC обычно имеет переменную длину инструкции и смешанные операнды.
RISC обычно имеет фиксированный формат и 2-3 регистровых операнда.
CISC удобнее для плотного кода.
RISC удобнее для регулярной реализации и конвейера.

### Схема / картинка

![CISC vs RISC instruction](../assets/course-fig/cisc-risc-instruction.png)

Схема показывает одну сложную CISC-инструкцию и серию простых RISC-инструкций.

### Подробный ответ

Архитектурное сравнение — это сравнение видимой программисту ISA.

**CISC:**

- инструкции могут быть сложными;
- длина инструкции переменная;
- операнды могут быть смешанными: регистры, память, непосредственные значения;
- арифметика может обращаться к памяти;
- код плотнее и ближе к высокоуровневому действию.

**RISC:**

- инструкции проще и однообразнее;
- формат чаще фиксированный;
- арифметика работает по регистрам;
- доступ к памяти вынесен в `load`/`store`;
- код может быть длиннее, но легче анализируется и декодируется.

С точки зрения архитектуры CISC стремится сделать мощными сами команды.
RISC стремится сделать команды регулярными, чтобы их поток исполнялся быстрее.

### Ключевые определения

- **CISC** — архитектура со сложными инструкциями.
- **RISC** — архитектура с регулярными простыми инструкциями.
- **Load-store** — архитектурное разделение памяти и регистровых операций.

### Пример

CISC может записать `mul R1 <- A * B` с доступом к памяти.
RISC сначала загрузит `A` и `B` в регистры, затем выполнит `mul`.

### Возможные вопросы преподавателя

- Почему CISC код плотнее?
- Почему RISC код регулярнее?
- Что значит "операнды смешанные"?

### Что обязательно запомнить

- CISC: сложные команды и плотный код.
- RISC: простые команды и регулярность.
- Главное архитектурное отличие — работа с памятью и формат инструкций.

## Билет 12. Как сравниваются CISC и RISC с точки зрения микро-архитектуры?

### Короткий ответ

С точки зрения микроархитектуры CISC сложнее декодировать и труднее конвейеризировать.
Его инструкции имеют разную длину, длительность и доступ к памяти.
Control Unit часто усложняется или реализуется микрокодом.
RISC проще декодировать, потому что инструкции однообразны.
Место сложного декодера и микрокода можно отдать регистрам и кешу.
RISC лучше подходит для pipeline.
Современные CISC-процессоры могут внутри использовать RISC-подобное ядро.

### Схема / картинка

![Типовая организация RISC](../assets/course-fig/proc-risc-typical-stages.png)

Схема показывает регулярную организацию RISC-процессора по стадиям.

### Подробный ответ

Микроархитектура — это то, как ISA реализована внутри.

Для CISC сложность возникает из-за нерегулярности:

- переменная длина инструкций;
- разные форматы;
- разные длительности;
- память может участвовать в арифметике;
- сложные режимы адресации.

Поэтому декодирование и управление часто требуют микрокода.
Конвейер строить трудно: не все инструкции удобно разбить на одинаковые стадии.

Для RISC ситуация проще:

- инструкции фиксированнее и однообразнее;
- декодер проще;
- операции с памятью отделены;
- регистровые операции похожи друг на друга;
- стадии конвейера легче балансировать.

Именно поэтому RISC в лекции связывается с pipeline.
В обсуждении также отмечено, что сложную CISC ISA часто проще исполнять через внутреннюю трансляцию к RISC-подобным операциям.

### Ключевые определения

- **Микроархитектура** — внутренняя реализация ISA.
- **Декодер** — блок разбора инструкции.
- **RISC-подобное ядро** — внутренняя регулярная реализация сложной внешней ISA.

### Пример

Конвейер для простой RISC ISA можно сделать понятным по стадиям.
Для M68K/CISC похожий конвейер ограничен сложной адресацией и разными форматами.

### Возможные вопросы преподавателя

- Почему RISC проще конвейеризировать?
- Почему CISC часто использует микрокод?
- Зачем CISC-процессору RISC-ядро внутри?

### Что обязательно запомнить

- CISC сложнее внутри.
- RISC регулярнее внутри.
- Конвейер любит однообразные стадии.
- Внешняя ISA и внутренняя микроархитектура могут различаться.

### Проверить

- `[проверить]` Внутренняя трансляция CISC в RISC-подобные операции дана устно, без подробной схемы.

## Билет 13. Каковы особенности кодирования инструкций при сравнении CISC и RISC?

### Короткий ответ

CISC обычно использует более сложное и переменное кодирование инструкций.
Инструкция может иметь разную длину и разное число операндов.
Поля могут описывать opcode, режимы адресации, память, регистры и immediate.
Это повышает плотность кода.
RISC стремится к единому формату и фиксированной длине.
Операнды чаще являются регистрами.
Простое кодирование ускоряет декодирование.
Цена RISC — больше инструкций для сложного действия.

### Схема / картинка

![CISC vs RISC instruction](../assets/course-fig/cisc-risc-instruction.png)

Картинка иллюстрирует разницу: одна сложная инструкция против нескольких простых.

### Подробный ответ

CISC-кодирование отражает богатую ISA.
Инструкция может быть короткой или длинной.
Она может включать opcode, несколько операндов, режимы адресации и непосредственные значения.
Плюс — высокая плотность кода: одно машинное слово или короткая последовательность может описать много работы.
Минус — декодеру нужно разбирать много вариантов.

RISC-кодирование отражает регулярность.
Инструкции обычно имеют фиксированный размер.
Форматов мало.
Операнды чаще указывают регистры, а работа с памятью вынесена в отдельные инструкции.
Плюс — декодер проще и быстрее.
Минус — выражение сложной операции требует нескольких команд.

Именно кодирование связано с pipeline.
Если процессор должен каждый такт выбирать новую инструкцию, ему удобно, когда инструкция помещается в машинное слово и быстро декодируется.

### Ключевые определения

- **Фиксированная длина инструкции** — одинаковый размер команд.
- **Переменная длина инструкции** — размер зависит от opcode, операндов и адресации.
- **Плотность кода** — полезная работа на объём машинного кода.

### Пример

RISC может заменить одну CISC-инструкцию четырьмя командами `load`, `load`, `op`, `store`.
Кода больше, но каждая команда проще.

### Возможные вопросы преподавателя

- Почему фиксированная длина помогает выборке инструкций?
- Почему CISC код плотнее?
- Как immediate влияет на длину инструкции?

### Что обязательно запомнить

- CISC кодирует много вариантов.
- RISC кодирует просто и регулярно.
- CISC плотнее.
- RISC легче декодировать и конвейеризировать.

## Билет 14. Каковы принципы конвейерного исполнения инструкций? Как это влияет на загрузку и производительность?

### Короткий ответ

#### Каковы принципы конвейерного исполнения инструкций?

Конвейер разбивает выполнение инструкции на несколько стадий.
Разные инструкции одновременно находятся на разных стадиях.
Один такт соответствует одной стадии.

#### Как это влияет на загрузку и производительность?

После заполнения конвейера процессор может завершать инструкцию каждый такт.
Загрузка ресурсов растёт, потому что разные блоки работают параллельно.
Отдельная инструкция может не стать быстрее.
Но поток инструкций выполняется быстрее.
Производительность падает, если появляются конфликты и пузырьки.

### Схема / картинка

![Pipeline](../assets/course-fig/proc-risc-pipeline.jpg)

Схема показывает стадии конвейера и движение инструкций по ним.

### Подробный ответ

#### Каковы принципы конвейерного исполнения инструкций?

Конвейер работает как производственная линия.
Инструкция не исполняется одним неделимым действием.
Она проходит стадии: выборка, декодирование, исполнение, память, запись результата.

Чтобы построить конвейер, нужно:

- выделить стадии;
- перестроить внутренние блоки процессора под эти стадии;
- соединить стадии регистрами;
- сбалансировать задержки комбинационной логики;
- каждый такт подавать новую инструкцию;
- каждый такт забирать результат после заполнения конвейера;
- разрешать конфликты между инструкциями.

#### Как это влияет на загрузку и производительность?

Без конвейера блоки процессора простаивают: пока одна инструкция исполняется, другие стадии не заняты.
С конвейером выборка одной инструкции, декодирование другой и исполнение третьей идут одновременно.
Поэтому загрузка аппаратных блоков растёт.

Производительность растёт по потоку.
Одна инструкция может проходить больше стадий и не стать быстрее по задержке.
Но завершённые инструкции появляются чаще.
Если возникает конфликт, конвейер приходится задерживать, и выигрыш уменьшается.

### Ключевые определения

- **Pipeline** — организация выполнения через стадии.
- **Стадия конвейера** — часть выполнения инструкции за один такт.
- **Заполнение конвейера** — начальный период, пока стадии ещё не заняты полностью.

### Пример

Две инструкции без конвейера проходят стадии последовательно, а с конвейером вторая начинает выборку до завершения первой.

### Возможные вопросы преподавателя

- Почему конвейер повышает throughput, а не обязательно latency?
- Почему стадии нужно балансировать?
- Почему конфликты ломают идеальную производительность?

### Что обязательно запомнить

- Разбиваем исполнение на стадии.
- Разные инструкции идут одновременно.
- После заполнения можно получать результат каждый такт.
- Конфликты снижают выигрыш.

### Проверить

- `[проверить]` В источниках приведены учебные схемы; реальные конвейеры могут быть глубже.

## Билет 15. Каковы типовые стадии RISC процессора?

### Короткий ответ

Типовые стадии RISC процессора: Instruction Fetch, Instruction Decode, Instruction Execute, Memory Access и Write Back.
Instruction Fetch читает инструкцию по счётчику команд.
Instruction Decode декодирует инструкцию и читает регистры.
Instruction Execute выполняет операцию или готовит адрес.
Memory Access читает или пишет память.
Write Back записывает результат в регистр.
Такая разбивка удобна для load-store архитектуры.
Она делает поток инструкций регулярным.

### Схема / картинка

![Типовые стадии RISC](../assets/course-fig/proc-risc-pipeline.jpg)

Схема перечисляет пять классических стадий RISC-конвейера.

### Подробный ответ

**Instruction Fetch (IF).**
Процессор читает следующую инструкцию по `PC`.
Для конвейера важно делать это каждый такт.

**Instruction Decode (ID).**
Инструкция декодируется.
Процессор понимает opcode, формат и читает нужные регистры.

**Instruction Execute (EX).**
Выполняется арифметическая или логическая операция.
Для memory reference здесь может готовиться адрес.

**Memory Access (MEM).**
Процессор обращается к памяти.
Команды `load` читают данные, `store` записывают.

**Write Back (WB).**
Результат записывается обратно в регистровый файл.

Эти стадии характерны для RISC, потому что инструкции регулярны.
Если операция сложная и нерегулярная, разложить её на такие стадии труднее.

### Ключевые определения

- **IF** — выборка инструкции.
- **ID** — декодирование инструкции и чтение регистров.
- **EX** — выполнение операции.
- **MEM** — доступ к памяти.
- **WB** — запись результата.

### Пример

Для `load R3 <- [addr]`: IF читает инструкцию, ID декодирует, EX готовит адрес, MEM читает память, WB пишет значение в `R3`.

### Возможные вопросы преподавателя

- На какой стадии читаются регистры?
- На какой стадии обращаются к памяти?
- Почему `load` использует MEM и WB?

### Что обязательно запомнить

- IF, ID, EX, MEM, WB.
- Пять стадий — базовая учебная модель.
- Load-store хорошо подходит под такую разбивку.

### Проверить

- `[проверить]` В источниках также упоминаются операции с разной латентностью; билет даёт базовую модель.

## Билет 16. Какие проблемы и архитектурные ограничения связаны с конвейеризацией?

### Короткий ответ

Проблемы конвейеризации связаны с тем, что инструкции не независимы.
Есть структурные конфликты из-за общих ресурсов.
Есть конфликты по данным, когда следующая инструкция ждёт результат предыдущей.
Есть конфликты по управлению из-за переходов.
Стадии должны быть сбалансированы по времени.
Инструкция должна быстро выбираться и декодироваться.
Нерегулярные CISC-инструкции труднее конвейеризировать.
Конвейер может снижать предсказуемость времени исполнения.

### Схема / картинка

![Проблемы конвейера](../assets/course-fig/proc-risc-typical-stages.png)

Схема помогает увидеть, почему стадии и ресурсы должны быть согласованы.

### Подробный ответ

Конвейер работает идеально только если каждая инструкция независима и каждая стадия доступна.
В реальных программах это не так.

**Структурные конфликты.**
Несколько стадий хотят один ресурс.
Пример: выборка инструкции и доступ к данным требуют одну память.

**Конфликты по данным.**
Одна инструкция должна использовать результат другой, но результат ещё не записан.
Типичный случай — RAW.

**Конфликты по управлению.**
Переходы мешают заранее знать следующую инструкцию.
Эта тема в lecture-09 обозначена, а подробнее уходит в следующий блок.

**Архитектурные ограничения.**
Инструкции желательно иметь простыми, регулярными и фиксированного формата.
Нужно каждый такт загружать новую инструкцию.
Стадии должны быть примерно одинаковыми по длительности.
Если одна стадия медленная, весь конвейер работает по ней.

### Ключевые определения

- **Structural dependency** — конфликт аппаратного ресурса.
- **Data hazard** — зависимость по данным между инструкциями.
- **Control hazard** — неопределённость следующей инструкции из-за перехода.
- **Bubble** — пустой такт/элемент для задержки конвейера.

### Пример

Если одна инструкция читает память на стадии MEM, а другая в тот же такт хочет выбрать инструкцию на стадии IF из той же памяти, возникает структурный конфликт.

### Возможные вопросы преподавателя

- Почему инструкции должны быть регулярными?
- Почему CISC сложнее конвейеризировать?
- Почему переходы мешают выборке следующей инструкции?

### Что обязательно запомнить

- Три типа конфликтов: structural, data, control.
- Нужны регулярные стадии.
- Самая медленная стадия задаёт темп.
- Конвейер повышает throughput, но создаёт новые проблемы.

### Проверить

- `[проверить]` Конфликты по управлению подробно относятся к lecture-10.

## Билет 17. Как разрешаются структурные конфликты в конвейере?

### Короткий ответ

Структурные конфликты возникают, когда стадиям нужен один и тот же аппаратный ресурс.
Пример — общий доступ к памяти для IF и MEM.
Полное решение — дать ресурсам независимость.
Для памяти это Гарвардская архитектура или двухпортовая память.
Если ресурс нельзя разделить, вставляют пузырёк.
Пузырёк задерживает поток и даёт ресурсу освободиться.
Это просто, но снижает производительность.

### Схема / картинка

```text
| I1 | IF | ID | EX | MEM | WB |
| I4 |    |    |    | IF  | ID |
```

В конфликтном такте стадия `IF` и стадия `MEM` требуют общий ресурс памяти.

### Подробный ответ

Структурный конфликт — это конфликт аппаратуры.
Процессор хотел бы выполнить комбинацию стадий, но ресурсов не хватает.
В lecture-09 главный пример — единая память команд и данных.

Решения:

**Гарвардская архитектура.**
Память инструкций и память данных разделены.
IF ходит в память инструкций, MEM ходит в память данных.
Конфликт исчезает физически.

**Двухпортовая память.**
Один блок памяти имеет два независимых порта.
Это позволяет двум обращениям идти параллельно.
Минус — площадь и сложность памяти.

**Bubble.**
Если ресурс нельзя удвоить, конвейер задерживают.
Вместо следующей инструкции запускают пустой элемент, чтобы конфликтная стадия подождала.
Это проще, но уменьшает throughput.

### Ключевые определения

- **Структурный конфликт** — конфликт за общий аппаратный ресурс.
- **Двухпортовая память** — память с двумя независимыми портами доступа.
- **Bubble** — пустой элемент конвейера.

### Пример

При общей памяти `I1` на стадии MEM читает данные, а `I4` на стадии IF хочет выбрать инструкцию.
Один порт памяти не может обслужить оба запроса.

### Возможные вопросы преподавателя

- Почему Гарвардская архитектура решает этот конфликт?
- Почему двухпортовая память дорогая?
- Почему bubble снижает производительность?

### Что обязательно запомнить

- Структурный конфликт — не хватает аппаратуры.
- Полные решения: разделить ресурс или сделать многопортовым.
- Простое решение: bubble.

- `[проверить]` В билете текстовая схема упрощена; полная таблица есть в слайдах.

## Билет 18. Как разрешаются конфликты по данным в конвейере?

### Короткий ответ

Конфликты по данным возникают, когда инструкция зависит от результата другой инструкции.
Главный пример — RAW: чтение после записи.
Первая инструкция ещё не записала результат, а вторая уже хочет его читать.
Разрешать такие конфликты можно перестановкой инструкций.
Можно переименовать регистры, если зависимость ложная.
Можно вставить пузырёк и подождать.
Можно сделать forwarding и передать результат между стадиями напрямую.

### Схема / картинка

![Data hazard](../assets/course-fig/pipeline-data-hazard.png)

Схема показывает RAW-конфликт: следующая инструкция хочет использовать значение до записи результата.

### Подробный ответ

Конфликт по данным — это зависимость между инструкциями.
В lecture-09 перечислены варианты:

**RAW: Read after Write.**
Инструкция читает регистр, который предыдущая инструкция ещё не успела записать.
Это настоящий data dependency.

**WAR: Write after Read.**
Запись после чтения может мешать перестановке инструкций.

**WAW: Write after Write.**
Две записи в одно место могут стать проблемой при переупорядочивании или в особых случаях.

**RAR: Read after Read.**
Обычно не проблема, потому что чтение пассивной памяти состояние не меняет.

Способы разрешения:

- **перестановка инструкций** — компилятор или процессор ставит между зависимыми командами полезные независимые команды;
- **переименование регистров** — ложная зависимость исчезает, если записывать в другой физический регистр;
- **bubble** — зависимая инструкция ждёт;
- **forwarding** — результат передаётся напрямую между стадиями.

### Ключевые определения

- **RAW** — чтение после записи, настоящая зависимость по данным.
- **WAR** — запись после чтения.
- **WAW** — запись после записи.
- **RAR** — чтение после чтения.
- **Register renaming** — устранение ложных зависимостей заменой физического регистра.

### Пример

```asm
and r1 <- __ & __
sub __ <- r1 - __
```

`sub` хочет прочитать `r1`, но `and` ещё не записал новый результат.

### Возможные вопросы преподавателя

- Почему RAW самый неприятный?
- Почему RAR обычно не конфликт?
- Когда помогает переименование регистров?

### Что обязательно запомнить

- Data hazard — зависимость по данным.
- RAW — главный реальный конфликт.
- Решения: reorder, rename, bubble, forwarding.

### Проверить

- `[проверить]` WAR/WAW в источниках объяснены кратко и с оговорками.

## Билет 19. Как происходит разрешение конфликтов при помощи пузырька в конвейере?

### Короткий ответ

Пузырёк — это пустой элемент конвейера.
Он занимает стадии, но не выполняет полезного действия.
Пузырёк вставляют, когда следующую инструкцию нельзя безопасно пустить дальше.
Он даёт ресурсу освободиться или результату успеть записаться.
Это не то же самое, что `NOP`, потому что `NOP` является настоящей инструкцией.
Пузырёк проще реализовать, чем сложную оптимизацию.
Но он снижает производительность, потому что стадии простаивают.

### Схема / картинка

![Пузырёк в конвейере](../assets/course-fig/pipeline-data-forwarding-with-bubble.png)

Картинка показывает вариант, где ожидание совмещается с дальнейшим пробросом результата.

### Подробный ответ

Bubble вставляется, когда конвейер должен подождать.
Он проходит по стадиям как пустая работа.
В таблицах слайдов он обозначен `0`.

При структурном конфликте bubble даёт время общему ресурсу.
Например, если память занята стадией MEM, выборку следующей инструкции можно задержать.

При конфликте по данным bubble даёт время предыдущей инструкции записать результат.
Зависимая инструкция стартует позже и читает уже корректное значение.

Важно отличие от `NOP`.
`NOP` — настоящая инструкция ISA.
Её надо выбрать из памяти, декодировать и провести по модели исполнения.
Bubble — внутренняя задержка конвейера, а не полноценная команда программы.

Плюс bubble — простота.
Минус — потеря тактов и снижение загрузки конвейера.

### Ключевые определения

- **Bubble** — пустой элемент конвейера.
- **Stall** — остановка или задержка продвижения конвейера.
- **NOP** — настоящая инструкция, которая ничего не делает.

### Пример

Если `sub` зависит от результата `and`, можно вставить bubble между ними, чтобы `and` успел записать `r1`.

### Возможные вопросы преподавателя

- Почему bubble не равен `NOP`?
- Почему bubble простой, но медленный?
- В каких конфликтах bubble помогает?

### Что обязательно запомнить

- Bubble занимает стадии без полезной работы.
- Он нужен для ожидания.
- Он снижает throughput.
- `NOP` — команда, bubble — внутренняя пауза.

### Проверить

- `[проверить]` В источниках разные таблицы показывают bubble для структурного и data conflict; билет объединяет принцип.

## Билет 20. Как происходит разрешение конфликтов при помощи проброса операндов в конвейере?

### Короткий ответ

Проброс операндов, или forwarding, передаёт результат между стадиями конвейера напрямую.
Он минует запись в регистровый файл и последующее чтение из него.
Это помогает при RAW-конфликте.
Например, результат первой инструкции из EX можно сразу подать на EX следующей.
Если результат появляется достаточно рано, bubble не нужен.
Если результат появляется поздно, forwarding сочетают с одним пузырьком.
Так ожидание короче, чем при полном ожидании записи в регистр.

### Схема / картинка

![Operand forwarding](../assets/course-fig/pipeline-data-forwarding.png)

Схема показывает передачу результата из одной стадии конвейера в другую без прохода через регистровый файл.

### Подробный ответ

Forwarding решает проблему "результат уже вычислен, но ещё не записан туда, откуда его обычно читает следующая инструкция".
В обычной схеме следующая инструкция ждала бы стадии WB.
При forwarding процессор добавляет специальные линии между стадиями.

Пример:

- инструкция `i1` вычисляет результат;
- инструкция `i2` должна использовать его как операнд;
- результат `i1` передаётся с поздней стадии `i1` прямо на стадию EX у `i2`;
- регистровый файл обходится.

Это работает, если значение уже существует к нужному такту.
Если значение появляется слишком поздно, например после Memory Access, одного forwarding недостаточно.
Тогда ставят bubble, а затем пробрасывают значение.
Это всё равно лучше, чем ждать до полного завершения инструкции.

### Ключевые определения

- **Forwarding** — прямой проброс результата между стадиями конвейера.
- **Bypassing** — другое название forwarding.
- **Operand forwarding** — проброс операнда для зависимой инструкции.
- **Регистровый файл** — обычное место записи и чтения регистровых результатов.

### Пример

```asm
and r1 <- __ & __
sub __ <- r1 - __
```

Результат `and` можно пробросить в `sub`, не дожидаясь записи `r1` на стадии WB.

### Возможные вопросы преподавателя

- Почему forwarding помогает именно RAW?
- Почему иногда нужен forwarding плюс bubble?
- Почему forwarding усложняет DataPath?

### Что обязательно запомнить

- Forwarding передаёт результат напрямую.
- Он обходит регистровый файл.
- Он уменьшает число пузырьков.
- Если результат поздний, нужен bubble + forwarding.

### Проверить

- `[проверить]` Конкретные пути forwarding зависят от микроархитектуры.
